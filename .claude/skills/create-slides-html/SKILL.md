# reveal.js 高品質HTMLスライド作成スキル

reveal.js + Tailwind CSS(CDN) で、AIっぽさを消したHTMLスライドを作る。**全プロセス**を定義する。デザインに入る前に、必ずフェーズ0・1・2・2.5を完了すること。なぜなら、見た目のルールだけ整えても、コンテンツの設計と文章がAIっぽいままだと「機械が作った資料」に見えてしまうからだ。

成果物は原則として**単一のHTMLファイル**（reveal.js と Tailwind を CDN から読み込む）。`slide-template/` のようなビルド構成は理想形の参考であって、毎回そこまで作る必要はない。ユーザーが既存テンプレートやリポジトリを指定した場合だけ、その構成に合わせる。

---

## フェーズ0：参照デザインの検出とトークン抽出（最初に必ず実施）

スライド依頼を受けたら、**まず参照デザインが指定されているか**を確認する。

判定（いずれかに該当すれば参照ありとして扱う）:
- ユーザーが `.pptx` のパスを指定（「このpptxと同じデザインで」等）
- ユーザーが参照したいWebサイトのURLを指定（「このサイトと同じ雰囲気で」等）
- 会話中で既存資料・既存サイトを「同じ見た目で」「テンプレとして」と指示された

分岐:

**参照PPTXがある場合** → `create-design-md-from-pptx <pptxパス>` を実行して DESIGN.md を生成させ、読み込む。
**参照サイトがある場合** → `create-design-md <URL>` を実行して DESIGN.md を生成させ、読み込む。

参照ありの場合、生成された DESIGN.md の Color Palette / Typography / Components / Layout / Do's & Don'ts を、**フェーズ3のデフォルトデザインシステムの代わりに** Tailwind config（`c-` クラス・カラー定義）へ落とし込む。デフォルトの配色制約は参照元のパレットに置き換える。

**参照がない場合** → フェーズ3のデフォルトデザインシステム（`references/design.md`：Howard由来）を Tailwind config に定義して使う。

重要：参照が与えられているのにデフォルトのデザインで作らないこと。ただし**フェーズ1・2・2.5はどちらのケースでも必ず実施する**。これらはコンテンツ設計のフェーズで、見た目とは独立している。

## フェーズ1：ヒアリング（スライド着手前に必ず実施）

すでに会話で明示されている項目はスキップしてよい。不足していれば確認する。

① **読み手は誰か** — 役職・立場、前提知識レベル、意思決定権の有無
② **このスライドで達成したいこと** — 読み手に何をしてほしいか（承認・理解・行動）、成功の定義
③ **全体の結論（So What）** — 全部見た後に残るべき1文。未定なら一緒に言語化する
④ **情報の流れ** — 結論先行（PREP）か、課題から始める順か

---

## フェーズ2：レポート作成と確認（スライド変換前のゲート）

ヒアリング完了後、スライドを作る前にレポート形式で内容を整理し、ユーザーに確認を求める。

```
# [タイトル]

## 対象読者と目的
- 読み手：[役職・前提知識]
- 達成すること：[具体的なゴール]
- 取ってほしいアクション：[承認・理解・行動など]

## 全体結論（エグゼクティブサマリー）
[全部見た後に残すべき1〜3文]

## 構成（起承転結）
### 起：前提・背景
### 承：現状・課題
### 転：示唆・分析
### 結：結論・提言・次のアクション

## スライド構成案（目次）
1. [スライドタイトルとメッセージライン案]
2. ...

## 懸念点・要確認事項
- [不足データやユーザー確認が必要な点]
```

提示後、必ず確認する：「上記の内容・構成でよろしいですか？修正があればお知らせください。承認いただけたらスライド作成に進みます。」 ユーザーの承認なしにスライド作成へ進まない。

## フェーズ2.5：テキストのAI臭除去（スライド変換前に必ず実施）

レポート承認後、スライドに載せるテキストを書き直す。文章からAIっぽさを消すことで、人が作ったスライドに仕上がる。

- **カギ括弧の多用を禁止** — 強調や定義で囲まず文脈に溶かす。固有名詞や引用など本当に必要なときだけ最小限に。
  NG「考える工程」と「形にする工程」を分ける → OK 考える作業と形にする作業を分ける
- **丸括弧で補足を逃がさない** — 括弧に入れるほどの説明なら本文に組み込む。不要なら削る。
  NG ターゲット（誰に見せるのか）が不明確 → OK 誰に見せるのかが不明確
- **コロン＋ラベル列挙を避ける** — 表の列ヘッダー等を除き、コロンでラベルを並べない。
- **テンプレ的な見出しを使わない** — 「3つの共通課題がある」「最適解である」を避け、具体的な動詞で言い切る。
- **抽象語だけで押し切らない** — 「重要」「効果的」「最適」「本質」だけで済ませず、何がどうなるかを動詞で書く。
- **定型の締め文句を入れない** — 「お気軽にお声がけください」「参考になれば幸いです」は使わない。
- **同義語の言い換え連打をしない** — 重要・大切・欠かせない、と繰り返さず1回で言い切る。
- **文のリズムを均一にしない** — メッセージラインが全スライド同じ構文にならないよう変化をつける。同じ型が3枚以上続かないように。

手順：承認レポートのテキストを上記で書き直し、カギ括弧の数を数え、メッセージラインを通しで読んでテンプレ感や同じ構文の連続がないか確認する。

## フェーズ3：reveal.js + Tailwind デザインシステム（承認後に適用）

> 参照PPTX/サイトから DESIGN.md を抽出した場合は、色・フォント・コンポーネントの値をその DESIGN.md で読み替える。以下の配色・フォントは**参照なしのデフォルト**で、`references/design.md`（Howard由来のデザインシステム）に準拠する。

実装の構造ルールとサンプルは `references/structure.md`、デフォルトのデザイントークン（配色・フォント・コンポーネント）は `references/design.md` に詳細がある。**スライドのHTMLを書く前に、必ず `references/structure.md` と `references/design.md` を読むこと。** ここでは要点だけ示す。

### 3-1. 出力形式

- 単一HTMLファイル。reveal.js と Tailwind は CDN から読み込む（`assets/base.html` を雛形として使う）。
- Tailwind の設定（カラー・フォント・`c-` カスタムクラス）は HTML 内の `tailwind.config` に書く。`references/structure.md` の `lib/tailwind/config.js` 相当をインラインで持つ。

### 3-2. 構造ルール（厳守。理由とともに守る）

- **日本語で記述する。**
- HTML構造は `.slides > section > section` のように **section を入れ子**にする（縦方向のスライドグループ）。
- **全 `<section>` の直下に r-stretch ラッパー**を置く。上ぞろえは `class="r-stretch grid"`、中央ぞろえは `class="r-stretch grid place-items-center"`。r-stretch は reveal.js が高さを伸縮させる仕組みで、これに乗せないと縦位置が崩れる。
- `<section>` に `padding` や `margin` を**つけない**。余白はラッパー側で取る。section に余白を付けると reveal.js のスケーリングと干渉する。
- **r-stretch と同じ要素に `gap` をつけない**。gap は子要素の高さ計算を壊すため、内側にもう1段 div を噛ませてそこで gap を使う。
- カスタムクラスは **`c-` プレフィックス**。定義は Tailwind config（HTML内の `tailwind.config` のインライン、構成を作る場合は `lib/tailwind/config.js`）にまとめる。
- **カラーコードを原則使わない**。`#3D698F` などを直書きせず、Tailwind config で定義した色名（`bg-slate` 等）を使う。
- **`ul` `ol` `li` は禁止**。箇条書きや階層は `div` で組む。reveal.js のリスト見た目がAIっぽさの元になりやすいため。
- **絵文字はユーザーが要求したときだけ**使う。

### 3-3. デフォルトのカラーパレット（Howard由来。Tailwind config に定義）

| 役割 | 色名(config) | 値 | RGB |
|------|--------------|-----|-----|
| 背景・余白 | `bg` | `#FFFFFF` | 255,255,255 |
| カード淡灰 | `surface` | `#F2F2F2` | 242,242,242 |
| 見出し・本文インク | `ink` | `#44546A` | 68,84,106 |
| 補助テキスト | `sub` | `#999999` | 153,153,153 |
| 主アクセント（スレートブルー） | `slate` | `#3D698F` | 61,105,143 |
| ティールグレー | `tealgrey` | `#80A7A5` | 128,167,165 |
| セージグリーン | `sage` | `#ABBE92` | 171,190,146 |
| マウブ（限定） | `mauve` | `#775864` | 119,88,100 |

- グラデーションは**使わない**。背景・色面はすべてベタ塗り。表紙・セクション扉は `slate`（スレートブルー）一色＋白文字。
- 純黒 `#000000` のベタ使いは避け、インクは `#44546A`。差し色はスレートブルー＋セージ系に集約し、マウブ `#775864` は最小限。赤・橙・紫は使わない。

### 3-4. タイポグラフィ（Howard由来）

- 見出し・タイトル・数値は **Arial Bold（全大文字 / ALL CAPS）**、本文・リードは **Lato Light（サンセリフ）**。一部の小見出しに Poppins SemiBold を使ってもよい。和文フォールバックは游ゴシック系。
- 見出しと本文は同じサンセリフ系だが、**太字（Arial Bold）と細字（Lato Light）の対比**で階層を作る。セリフ体は使わない。
- セクション見出しは必ず全大文字。大きさの目安（`references/structure.md` の `c-` クラスに集約）：表紙タイトル 最大 → セクション見出し → 統計数値 → リード/本文。
- 本文色は `#44546A`、補助は `#999999`。詳細は `references/design.md` の Hierarchy 表を参照。

### 3-5. レイアウト原則

- **2系統を作り分ける** — 表紙・セクション扉は `bg-slate`（スレートブルー一色）＋白文字、本文スライドは白背景＋`#44546A` 見出し。
- **セクション見出し** — 各スライド上部に全大文字の太字見出し（Arial Bold）。
- **メッセージライン** — 全スライドに、そのスライドの結論を1文で言い切る見出し（断言形）。フェーズ2のメッセージライン案を使う。
- **階層は箇条書き記号を使わずロジックツリー型** — 第1階層（主論点、最大3）／第2階層（根拠、各3まで）／第3階層（補足）を `div` のインデントと文字サイズ・色で表現する。
- **統計・数値** — `surface`（淡灰）や `slate` の色面に Arial Bold の数値（色面上は白、白地はスレート）。
- **余白** — 広めに取り、1スライド1メッセージ。左揃えか中央揃えを統一。

### 3-6. 装飾ルール（禁止事項）

フラットを守る。グラデーション背景、強い影・光彩・ベベル・3D、過剰なアニメーション、装飾的な囲み罫線、`・▶◆` などの装飾記号、絵文字（要求時を除く）は使わない。角丸はほぼ使わず（図解の円バッジを除く）矩形は直角。レイヤー差はベタ塗り色面の重なりで表す。統計カードに付ける淡い影（`0 2px 8px rgba(68,84,106,0.08)`）までは許容。

### 3-7. グラフ・チャートは Chart.js を使う

棒・折れ線・円・ドーナツなどのグラフを載せるときは、**必ず Chart.js（CDN）で描く**。手書きのSVGや画像、CSSだけの擬似グラフは使わない。理由は、データを差し替えても体裁が崩れず、凡例や軸が自動で整い、後から保守しやすいから。

- データの系列色は**デザインシステムのパレットを使う**（デフォルトなら `#3D698F`(slate) / `#80A7A5`(tealgrey) / `#ABBE92`(sage)）。Chart.js の設定はJSなので色はHEXで書くが、これは config と同値で、本文HTML側のカラーコード直書き禁止の例外。
- 影・グラデ・3Dは付けない（フラットのまま）。装飾的なアニメーションは切る（`animation: false` 推奨）。
- グラフは1スライド1つを基本にし、上部にメッセージライン（断言形）、グラフ下や脇に出典・注釈を置く。
- canvas は高さを持つ `div` の中に置く（`r-stretch` 直下に裸の canvas を置かない）。実装の雛形は `references/structure.md` の「§8 Chart.js」を参照。

### 3-8. 図解（ダイアグラム）の作法

概念図・フロー図は、グラフ（§3-7）と違い `div` や SVG で組む。次を守る。

- **図は「形が概念を表す」ものにする。** 関係に図の種類を合わせる：包含・部分↔全体＝同心円や入れ子、流れ・時系列＝矢印でつないだ箱、強弱・量＝大きさや位置。幅や色を変えただけのものを「ピラミッド」等と称さない（形が意味を持たないものは図ではない）。
- **対等でない関係を、対等な横並びカードで並べない。** 階層・包含・別カテゴリの関係があるなら、横並び3カードで済ませず、図でその関係を示す。
- **図を文章で実況しない。** 「〜の中に〜が入り…」のように図を見れば分かることを本文でなぞらない。図の読み方を説明するキャプション（「円の大きさが効く範囲です」など、自分の図のエンコードの解説）も本文に書かない。
- **担当・主体が分かれるフローは、ゾーン（囲み＋ラベル）や色で「誰がどこまでやるか」を示す。**
- 見出しと図で言い切れているなら、要約の一行を足さない。重複する締め文は、書き直すより消す。
