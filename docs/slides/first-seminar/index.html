Claudeについて 01 — 優位性
同じAI開発支援でも、Claudeを一番活かせるのは純正のClaude Code。
GitHub Copilot
Microsoft製アプリに組み込まれたAI
補完だけでなく、チャット・PRレビュー・CLI、Teams など Microsoft / GitHub の各所で使用できるが、性能はイマイチ
Cursor
モデルを選べるエディタ
AIコーディングエディタ。中で動かすモデルを切り替えられる（Claude や OpenAI 系など）。Claude は選択肢の一つ
Claude Code
Claude 純正のエージェント
Anthropic 製。Claude を最大限引き出し、仕事を渡すと手順を組んで最後まで進める
3つは役割が違います。
広く埋め込むCopilot、モデルを選べるCursorに対し、Claude CodeはClaude純正だから最新機能をそのまま使え、エージェントとして仕事を任せきれます。
Copilotは補完だけではない点に注意。MS/GitHubの各所（チャット・レビュー・Teamsなど）にAIを広く埋め込む。CursorはAI前提エディタで中のモデルを選べる（Claude/OpenAI系）、Claudeは選択肢の一つ。Claude CodeはAnthropic純正でClaudeをフルに活かすエージェント。優位性は範囲の広さではなく、純正ゆえに最新機能をそのまま使えて任せきれること。Q3が説明できないに寄っていればここで差分を補う。

どれも同じClaude。できることが少しずつ違うので用途で選択しましょう。
8つを「話す・相談する」「その場で作る」「任せる・埋め込む」の3グループに分けて見せる。全部を覚える必要はなく、入り口はWeb、開発はCLI、組み込みはAPI、と用途で選べばよい。今日の主役はCLIのClaude Code。
Claudeについて 03 — モデル
モデルは、仕事の難易度とトークン消費量で選ぶ。
消費が大きいほどコストや利用上限の減りも速い。難所だけOpus、普段はSonnetでOK。
重さとトークン消費量の2軸で選ぶ。重いモデルほど賢いが消費が大きく、コストや利用上限の減りも速い。Haiku＝軽く消費小、Opus＝賢いが消費大、日常はSonnet。難所だけOpus、普段はSonnet/Haikuで節約。Sonnetを真ん中に大きく置いて既定の主力だと示す。

デザインの仕様書(DESIGN.md)が重要です。次のページではDESIGN.mdとは何かを説明します。
デザイン編で紹介するSkillは2つ、どちらも同列。create-design-mdで仕様書(DESIGN.md)を作り、create-slides-htmlでそれを形にする。間に挟まるDESIGN.mdが軸、という構図を最初に置く。ClaudeDesignやモックはこの流れの中で触れる。
デザイン編 — DESIGN.mdとは
デザインの決まりを実測値で書き出した仕様書が、DESIGN.md。
任天堂のサイト（nintendo.com/jp）
このURLを /create-design-md に読ませると、DESIGN.mdが生成される。
さらにこれを元にモックを作成させると...
生成されたDESIGN.md（抜粋）
# Color Palette
Nintendo Red #e60012
Red Dark     #ce181e
Ink Gray     #3c3c3c
Panel Gray   #f2f2f2
# Typography
本文・見出し  Roboto / ヒラギノ角ゴ
本文の基準  12px
# Radius / Shadow
角丸 2px（円形50%はアバター）
影 ほぼ無し（フラット）
色・フォント・角丸・影が、推測ではなく実測値で並ぶ
左に実在サイト（nintendo.com/jp）のスクショ、右にそこから生成したDESIGN.mdの抜粋を並べる。ポイントは、見た目の印象ではなくHEXやフォント名という実測値で書き出されること。これが次のモック・スライド生成の入力になる。実物はdesign/create-mock/DESIGN.mdに全文あり。
デザイン編 — 出力例（モック）
任天堂のサイト風の、架空のサービスのLPができあがる。
架空の人材管理SaaSのLP
DESIGN.mdの値がそのまま効く
ブランド赤 #e60012 をロゴとCTAに
ニュートラルなグレー基調で階層をつくる
角丸はほぼなし・影なしのフラット
中身は人材管理SaaSでも、見た目は nintendo.com/jp 由来のテイストで揃う
DESIGN.md→モックの実出力。nintendo.com/jpから抽出したDESIGN.md（赤・フラット・グレー基調）を、人材管理SaaSのLPに当てたモック。中身は別物でも見た目はNintendo由来で揃う＝デザインシステムは別コンテンツにも使い回せる、と見せる。実物はdesign/create-mock/index.html。
デザイン編 — なぜ仕様書を挟むか
同じDESIGN.mdを当てると、モックもスライドも見た目がそろう。
DESIGN.mdを挟むだけで、毎回デザインを言葉で指示し直さずに済みます。
抽出したテイストを、モックにもスライドにも使い回して再現できます。
このスライドも/create-slides-htmlで生成されています。
デザインとは見た目の決まりごと。それをDESIGN.mdという仕様書に固定しておくと、モックもスライドも同じ値を読み込むだけで見た目がそろう。毎回口で指示し直さない、使い回せる、が再現性のメリット。create-slides-htmlがトークン軽くローカルで動くので、ライブでも回しやすい。
業務自動化編 — 事例 01
AIニュースの収集、投稿を毎週自動で。
収集とファイル出力はClaude、Teams投稿はPower Automateが担当します。
ClaudeのTeams連携は限定的なので、別ツールと組み合わせる必要があります。
時系列：①収集→②MarkdownとHTMLを出力→Markdownは Obsidian へ蓄積（ナレッジ）／HTMLは Power Automate が受け取って③Teams投稿。投稿はClaudeではなくPower Automateが担う点を、ゾーン分けで強調。スケジュール起動と既存ツール（Obsidian/Power Automate/Teams）の組み合わせ。
業務自動化編 — 事例 02
請求書を読み取り、申請用Excelに転記。
請求書PDFとレートのスクリーンショットは証跡としてサブフォルダに残すので、あとから根拠を確認できます。
事例1の外向き共有に対し、内部の集計の例。時系列：①Claude in Chromeで請求書PDFをダウンロード・抽出→②JCBのレートで円換算＋レート画面をスクリーンショット→③Excelに追記→④人が金額を最終チェック。請求書PDFとレートのスクリーンショットはExcelと同じフォルダのサブフォルダに証跡として残す。抽出〜追記はClaude、最終確認だけ人。Claudeと人の線引きが、自動化を実務に乗せる肝。アイスブレイクの「見極めは人」と地続き。
